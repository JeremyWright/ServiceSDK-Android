Getting Started

Sample Application

You can find a sample application on GitHub. The app showcases how to integrate the SDK into an app.

Installing the Pilot Package

You can find the most recent Pilot package at https://salesforcesos.com. Select "SOS Quick Setup" and click the "Install" button.

Once the package has been installed, navigate to the ‘SOS Quick Setup’ app and complete Steps 1-4. The SOS Quick Setup app will walk you through:

  • Check Org Requirements
  • Adding the Presence Widget
  • Create New Service Presence Status
  • Create New Routing Configuration
  • Create New Queue
  • Update Permission Set
  • Create New SOS Deployment
  • Adding Auto Case Pop Logic and Compact Page Layout View

Step 4 provides you with important information that you will need to supply when you are create an SOS session.

SOS Session Information

When you create an SOS session, you need to provide three pieces of information:

  • Liveagent Pod - The liveagent server that will be used to communicate between the SDK and salesforce (e.g. xx-1.x.salesforce.com).
  • Organization ID - fifteen character identifier for your entire organization (e.g. 00Dxxxxxxxxxxxx).
  • Deployment ID - fifteen character deployment identifier used to connect an incoming request to a specific queue (e.g. 0NWxxxxxxxxxxxx).

A fourth piece of information should be included if you are using Auto Case Pop:

  • User Identifier - The identifier (typically email address) of a customer that is already authenticated in the app. SOS will pass the identifier to the Service Cloud where it is queried against the Contact Object. If there is a match, the customer contact will be appended to the Case.

Support

SOS for Android currently supports API version 15+. Support may change during our alpha/beta period.

Installation

Using Android Studio with Gradle

The SOS SDK is available via our maven repository on S3. Add the following configuration to your build.gradle file:

    
        repositories {
        mavenCentral()
        // ...
        maven { url 'http://s3.amazonaws.com/salesforcesos.com/android/maven/sos/sdk/release' }
        }

        // ...

        dependencies {
        // Use the most recent version of the SDK available. You can find the most recent version
        // at https://salesforcesos.com/#android-details
        compile 'com.salesforce.android:sos:0.21.1'
        }

Permissions

The SOS manifest file contains the following permission declarations:

    
        ...
        <uses-permission android:name="android.permission.RECORD_AUDIO"/>
        <uses-permission android:name="android.permission.INTERNET"/>
        <uses-permission android:name="android.permission.ACCESS_NETWORK_STATE"/>
        <uses-permission android:name="android.permission.MODIFY_AUDIO_SETTINGS"/>
        <uses-permission android:name="android.permission.READ_PHONE_STATE"/>
        ...

The Android build system will automatically merge these permissions into your application at build time.

Usage

Basic

To start SOS, you must make a call to the static SOS class from some activity. Once this is done, SOS will continue to run in other activities (via activity lifecycle callbacks) until it is explicitly killed.

    
        private void startSos () {
        SosOptions options = new SosOptions(
        getString(R.string.sos_la_endpoint),
        getString(R.string.sos_org_id),
        getString(R.string.sos_deployment)
        );

        Sos.session(this, options).start();
        }

Advanced

You can provide more advanced customization of the SOS session by using SosConfiguration

        
            protected void onCreate(Bundle savedInstanceState) {
            ..
            SosOptions options = new SosOptions(
            .. // same params as above
            );

            SosConfiguration config = SosConfiguration.builder()
            // Whether or not the session can ever have audio, this is useful to avoid
            // that nasty squealing that comes along with local development
            .audio(false)

            // Use the new radial agent view. In the future, this will be the default. This is
            // currently an experimental feature.
            .roundAgentVideo(false)

            // The agent video should be scaled to a dp value. In the future this will be the
            // default, currently experimental.
            .agentViewDpScaled(false)

            // Enable hot corners. Hot corners will animate the view to the nearest corner of the
            // screen. This will also save network bandwidth.
            .hotCorners(true)

            // whether or not the dialogs that describe session initialization states appear, when
            // off, the session will appear once it is connected, but you will be given no
            // information about the initialization progress while it is setting up.
            .waitUi(false)

            // Enable the Two-Way Video experience that enables your customers to connect face to
            // face with an agent. This requires the use of at least one camera, and as such at
            // least one camera must be enabled via configuration (mentioned below). You may also
            // configure the "defaultShareType" to be a camera, in which case the Two-Way video
            // experience will launch automatically at the start of a session.
            //
            // In order to use the two-way video feature, you must declare <uses-permission
            android:name="android.permission.CAMERA>"
            // in your AndroidManifest.xml
            //
            // Default value: false
            .twoWayVideo(true)

            // Set the share type to be used when a session starts.
            //
            // Available options:
            // SosShareType.ScreenSharing
            // SosShareType.FrontFacingCamera
            // SosShareType.BackFacingCamera
            //
            // Default value: SosShareType.ScreenSharing
            .defaultShareType(SosShareType.FrontFacingCamera)

            // Allow or disallow the front facing camera to be used during a Two-Way Video session.
            // If the device does not have a front camera, the next usable camera will be selected
            // or the session will revert to screen sharing.
            //
            // Default value: true
            .frontFacingCamera(true)

            // Allow or disallow the back facing camera to be used during a Two-Way Video session.
            // If the device does not have a back camera, the next usable camera will be selected or
            // the session will revert to screen sharing.
            //
            // Default value: true
            .backFacingCamera(true)

            // once all options are set, build the config object
            .build();

            // Instantiate an optional, custom SosDialogPresenter to be used during the session.
            SosDialogPresenter myDialogPresenter = new CustomDialogPresenter();

            Sos.session(this, options)
            .configuration(config)
            .dialogs(myDialogPresenter)
            .start();
            }

There are some more fine grained controls over the session that we offer. Please see SosConfiguration for more details.

Once the session is running, most API calls are made through Sos. See this class for more details.
